HBASE charts

Helm charts for launching HBASE daemons in a K8s cluster. The main entry-point chart is hbase-k8s, which is a uber-chart that specifies other charts as dependency subcharts. This means you can launch all HBASE components using hbase-k8s.

HBASE on K8s supports the following features:

  • Master high availability (HA): Master server is responsible for monitoring all RegionServer instances in the cluster, and is the interface for all metadata changes. Master crash will cause service outage. If run in a multi-Master environment, all Masters compete to run the cluster. If the active Master loses its lease in ZooKeeper (or the Master shuts down), then the remaining Masters jostle to take over the Master role. HBASE on K8s supports HA.
  • HBase REST API: The one way to access HBase is using the REST interface, which uses HTTP verbs to perform an action, giving developers a wide choice of languages and programs to use. Note that the Thrift and REST clients usually don’t run any other services services like DataNode or RegionServers to keep the load down, and responsiveness high, for REST interactions.
  • HBase Thrift Server: Thrift interface is the faster and more lightweight. Apache Thrift is a cross-platform, cross-language development framework. HBase includes a Thrift API and filter language. The Thrift API relies on client and server processes.

Here is the list of all charts.

  • hbase-k8s: main uber-chart. Launches other charts.
  • hbase-master-k8s: a statefulset and other K8s components for launching HBASE master daemons, which is responsible for monitoring all RegionServer instances in the cluster, and is the interface for all metadata changes. The chart supports Master high availability (HA).
  • hbase-region-k8s: a statefulset and other K8s components for launching HBASE RegionServer daemons, which is responsible for serving and managing regions. In a distributed cluster, a RegionServer runs on a DataNode.
  • hbase-config-k8s: a configmap containing HBase config files for HBASE.
  • hbase-restserver-k8s: a deployment and other K8s components for launching REST Server daemons, which uses HTTP verbs to perform an action, giving developers a wide choice of languages and programs to use. By default, we will launch one REST Servers.
  • hdfs-thriftserver-k8s: a deployment and other K8s components for launching Thrift Server daemons, which is a cross-platform, cross-language development framework. By default, we will launch two Thrift Servers.

Prerequisite

Requires Kubernetes 1.16+ as the master and regionserver are using ClusterFirstWithHostNet, which was introduced in Kubernetes 1.16

Usage

Basic

The HBASE daemons can be launched using the main hbase-k8s chart. First, build the main chart using:

xxxxxxxxxx
  $ helm dependency build charts/hbase/hbase-k8s

Then launch the main chart. Specify the chart release name say "my-hbase", which will be the prefix of the K8s resource names for the HBASE components.

xxxxxxxxxx
  $ helm install -n my-hbase charts/hbase/hbase-k8s

Wait for all daemons to be ready. Note some daemons may restart themselves a few times before they become ready.

xxxxxxxxxx
  $ kubectl get pod -l release=my-hdfs

Advanced options

Enable ingress for Rest Server

if you kubernetes cluster installed nginx-ingress, you can enable restServer ingress and Rest Server Ingress will be created. We can enable the restServer ingress option to the helm chart command:

xxxxxxxxxx
  $ helm install -n my-hbase charts/hbase/hbase-k8s  \
      --set "hbase-restserver-k8s.ingress=true"

Using an existing zookeeper quorum

By default, HBASE on K8s pulls in the zookeeper chart in the incubator remote repo (https://kubernetes-charts-incubator.storage.googleapis.com/) as a dependency and launhces zookeeper daemons. But your K8s cluster may already have a zookeeper quorum.

It is possible to use the existing zookeeper. We just need set a few options in the helm install command line. It should be something like:

xxxxxxxxxx
  $helm install -n my-hbase charts/hbase/hbase-k8s  \
    --set condition.subchart.zookeeper=false  \
    --set global.zookeeperQuorum=zk-0.zk-svc.default.svc.cluster.local:2181,zk-1.zk-svc.default.svc.cluster.local:2181,zk-2.zk-svc.default.svc.cluster.local:2181

Setting condition.subchart.zookeeper to false prevents the uber-chart from bringing in zookeeper as sub-chart. And the global.zookeeperQuorum option specifies the custom address for a zookeeper quorum. Use your zookeeper address here.

Using an existing hdfs cluster

By default, HBASE on K8s pulls in the hdfs chart in the incubator remote repo as a dependency and launhces hdfs daemons. But your K8s cluster may already have a hdfs cluster.

It is possible to use the existing hdfs. We just need set a few options in the helm install command line. It should be something like:

xxxxxxxxxx
  $helm install -n my-hbase charts/hbase/hbase-k8s  \
    --set condition.subchart.hdfs=false  \
    --set hbase-config-k8s.customHbaseConfig.hbaseSite.dfs.nameservices=hdfs-k8s \
    --set hbase-config-k8s.customHbaseConfig.hbaseSite.dfs.ha.namenodes.hdfs-k8s: nn0,nn1 \
    --set hbase-config-k8s.customHbaseConfig.hbaseSite.dfs.namenode.rpc-address.hdfs-k8s.nn0: namenode-hostname-01:8020 \
    --set hbase-config-k8s.customHbaseConfig.hbaseSite.dfs.namenode.rpc-address.hdfs-k8s.nn1: namenode-hostname-02:8020 \
    --set hbase-config-k8s.customHbaseConfig.hbaseSite.dfs.client.failover.proxy.provider.hdfs-k8s: org.apache.hadoop.hdfs.server.namenode.ha.ConfiguredFailoverProxyProvider

if hdfs cluster created configMap resource which it named hdfs-k8s-config and contain core-sit.xml and hdfs-sit.xml data. We also need set a few options in the helm install command line. It should be something like:

xxxxxxxxxx
  $helm install -n my-hbase charts/hbase/hbase-k8s  \
    --set condition.subchart.hdfs=false  \
    --set global.hdfsConfigMap.enable=true \
    --set global.hdfsConfigMap.name=hdfs-k8s-config

Setting condition.subchart.hdfs to false prevents the uber-chart from bringing in zookeeper as sub-chart. And the hbase-config-k8s.customHbaseConfig.hbaseSite option specifies the core-sit.xml and hdfs-sit.xml config.

Pinning master to specific K8s cluster nodes

Optionally, you can attach labels to some of your k8s cluster nodes so that master will always run on those cluster nodes. This can allow your HBASE client outside the Kubernetes cluster to expect stable IP addresses. 

xxxxxxxxxx
  $ kubectl label nodes YOUR-HOST-1 hbase-master-selector=hbase-master
  $ kubectl label nodes YOUR-HOST-2 hbase-master-selector=hbase-master

You should add the nodeSelector option to the helm chart command:

xxxxxxxxxx
  $ helm install -n my-hbase charts/hbase/hbase-k8s  \
     --set hbase-master-k8s.nodeSelector.hbase-master-selector=hbase-master \
     ...

Excluding regionserver from some K8s cluster nodes

You may want to exclude some K8s cluster nodes from regionserver launch target. For instance, some K8s clusters may let the K8s cluster master node launch a regionserver. To prevent this, label the cluster nodes with hbase-regionserver-exclude.

xxxxxxxxxx
  $ kubectl label node YOUR-CLUSTER-NODE hbase-regionserver-exclude=yes
