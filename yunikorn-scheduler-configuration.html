Yunikorn Scheduler 配置

Yunikorn核心调度程序配置有两个需要配置的独立区域。调度程序服务本身， 比如web服务端口等，以及队列配置 。 这两种配置之间的区别是基于以下两点提出的:

  • 职责分离
  • 动态与静态

调度程序配置主要是静态的。服务运行时，无需更改Web服务端口或调度策略。队列配置动态得多，可以在服务运行时更改。通过职责分离，可以允许集群管理员更改调度程序队列的配置，应该禁止管理员更改调度程序本身配置。

与核心调度程序配置分开的是，我们有一个或多个 shim 配置。我们目前无法预期调度程序及其 shim 的部署模型。 与k8s-shim类似的 shim 可以在相同的容器或节点中运行，但不能保证它会运行 。我们也不知道一个核心调度程序将使用的多少个shim 。还有可能同一个shim 会有多个实例在核心调度程序中运行。因此，Shim配置必须独立于核心调度程序配置。

Scheduler 配置

Scheduler 配置包括启动调度器和相关服务所需的所有配置。该配置由一个简单的键值对组成。启动服务的所有配置必须是此配置的一部分。调度程序配置必须排除与队列相关的配置。

当前标识的调度程序配置 ：

  • 主机绑定
  • 服务端口
  • Web主机绑定
  • Web服务端口
  • SSL配置
  • Shims 配置
  • SchedulerACL

Queue 配置

队列的定义

初始化服务后，调度程序将在启动时从提供的配置文件中加载队列的配置。如果没有提供队列配置，则调度程序应以简单的默认配置启动，该配置将执行详细记录的默认行为。根据kubernetes定义，此配置可以是 configMap， 但不能是CRD。

队列配置是动态的。更改队列配置一定不需要重新启动调度程序。应该通过调用基于GO的API，基于REST的API或通过更新配置文件来允许更改。通过API进行的更改必须保留在配置文件中。通过API进行更改不是高优先级的要求，并且可以推迟到更高版本。

队列配置使用 queues 属性配置，它是一棵树结构。树的基础是root队列。队列配置必须定义单个root队列。队列配置中定义的所有队列均被视为托管队列。

root队列反映了整个群集的资源。不允许在root队列上进行资源设置。root队列中可用的资源是根据群集中已注册的节点资源来计算的。如果在root队列上进行资源限制，则集群将被人为地限制为特定大小，或者期望不存在的可用资源。

树中层次结构中的队列由.点字符（ASCII 0x2E）分隔。这间接意味着队列名称本身不能包含点，因为它会干扰层次结构分隔符。配置中包含点的任何队列名称都将导致该配置无效。但是，我们必须允许放置规则使用基于点的输入创建队列。

并非所有队列都可用于向其提交申请。只能将应用程序提交到队列下面的队列。这些队列定义为树的叶子队列。包含其他队列或子队列的队列被称为父队列。

除root队列外，每个队列还必须有一个父队列。root队列不能有父队列，并且会自动定义为父队列类型。不区分大小写的标准队列名称在层次结构中必须唯一。因此，只能通过其完全限定的路径唯一地标识层次结构中的队列。这意味着在不同层次结构中允许使用具有相同名称的队列。例：

xxxxxxxxxx
root.companyA.development
root.companyB.development
root.production.companyA

在示例中，队列companyA和companyB是父队列。两个development队列都是叶子队列。companyA队列的第二个实例是叶队列，它与第一个实例无关，因为它是在层次结构的不同级别上定义的。

在配置中定义的队列将被分配一个队列类型。这可以基于在层次结构中定义队列的方式隐式指定，也可以通过将可选的父属性设置为队列定义的一部分而显式指定。默认情况下，将根据配置为所有队列分配其类型。只有一种情况需要覆盖这个自动过程，那就是将配置中的叶子标记为父级。用例是放置规则一部分。 在这种情况下，配置可用于仅为非托管队列定义父队列。

访问控制列表在提交权限和管理权限之间进行了划分。对队列的提交访问权限允许指定的用户或组将应用程序提交到队列。管理权限允许提交到队列以及进行管理操作。当前，管理操作仅限于终止应用程序和将应用程序移至其他队列。

从树的最低点开始，递归检查访问控制列表，直到树的根。换句话说，当队列的访问控制列表不允许访问时，将检查父队列。一直重复检查直到队列的根。

在除root队列之外的每个队列上，可以设置以下属性：

  • 队列类型：

    • Parent (boolean)

  • 资源设置：

    • Guaranteed (resource)
    • Maximum (resource)

  • 正在运行的应用程序限制：

    • Maximum (integer)

  • 队列权限：

    • SubmitACL（ACL）
    • AdminACL（ACL）

  • 抢占设置：

    • PreEmptionAllowed（boolean）

  • 应用排序算法：

    • ApplicationSortPolicy (枚举: fair, fifo)

在root队列上，只能设置以下属性：

  • 正在运行的应用程序限制：

    • Maximum (integer)

  • 队列权限：

    • SubmitACL（ACL）
    • AdminACL（ACL）

  • 应用排序算法：

    • ApplicationSortPolicy (枚举: fair, fifo)

用户定义

由用户运行的应用程序可以在一个或多个队列中运行。队列可以对使用的资源进行限制。这并不限制群集中用户可以使用的资源量。

从管理角度来看，设置特定用户可以使用的资源的限制可能很重要。在这种情况下，将用户广义地定义为提交申请的身份。这可以是服务，也可以是个人，从调度角度来看没有区别。用户限制可以防止行为不端的用户或应用程序接管队列或群集。从多租户的角度来看，用户限制还允许共享或细分租约中的资源，但是这是已定义的。

添加基于用户的限制将使集群管理员可以控制用户在集群范围内的资源使用情况：

  • 正在运行的应用程序限制：

    • Maximum (integer)

  • 资源设置：

    • Maximum (resource)

Placement 规则定义

调度程序可以动态地将应用程序放入队列中。这意味着在提交应用程序时不必包括要运行的队列。placement规则根据应用程的序详细信息将应用程序放置在队列中。规则将按照定义的顺序执行。

在规则的评估期间 ，结果可以是包含点的队列名称 。特别是对于符合POSIX的用户名和组名。当规则生成包含点的部分队列名称时，必须将其替换，因为它是层次结构中的分隔符。替换文字为_dot_

第一个匹配的规则（即返回完全限定的队列名）将停止规则的执行。如果应用程序未放在规则列表的末尾，则该应用程序将被拒绝。仅当规则允许创建队列时，规则才能返回配置中未定义的队列。

由placement规则创建的这些队列被视为非托管队列，因为它们在配置中不受管理员管理。管理员不能影响非托管队列的创建或删除。调度程序在需要时创建队列，在不再使用时会自动删除队列。

规则将提供标准队列名称作为结果。为了允许更深层的队列嵌套，可以将队列的父级设置为规则评估的一部分。规则定义应允许指定固定配置的完全合格的父级，也可以调用第二条规则来生成父级队列。默认情况下，会生成一个队列作为根队列的子级。

示例：user1(属于用户组user1和companyA的成员用户)提交的应用程序放置在基于UserName规则的队列中：

xxxxxxxxxx
Rule name: UserName
    Parent: root.fixedparent
Result: root.fixedparent.user1
​
Rule name: UserName
    Parent: SecondaryGroup
  Filter:
        Type: allow
      Groups: company.*
Result: root.companyA.user1
​
Rule name: UserName
Filter:
    Users: user2,user3
Result: denied placement

将应用程序放入队列的默认行为与使用在提交过程中提供的队列相同，这是一条规则，该规则将提供的队列中的create标志设置为false。

访问权限将作为规则评估的一部分而强制执行。对于托管队列，这意味着将检查队列本身的ACL。对于非托管队列，将检查父队列ACL。有关访问控制列表和检查的定义，请参见“ 访问控制列表”一章。

在配置中定义placement规则需要每个规则提供以下信息：

  • 名称：

    • Name (string)

  • 父级

    • Parent (string)

  • 创建标志：

    • Create (boolean)

  • Filter ：

    • 要应用规则的用户/组的正则表达式或列表。

Filter 可用于允许使用规则（默认行为）或拒绝使用规则。与过滤器匹配的用户或组将被允许或拒绝。过滤器定义如下：

  • Type ：

    • Type (string) ，不能有任何值（空）或“ allow”或“ deny”，不区分大小写。

  • Users ：

    • 零个或多个用户名的列表。如果列表正好是一个长列表，它将被解释为一个正则表达式。

  • Groups ：

    • 零个或多个组名的列表。如果列表正好是一个长列表，它将被解释为一个正则表达式。

建议的placement申请规则是：

  • Provided ：返回提交期间提供的队列
  • UserName ：返回用户名
  • PrimaryGroupName：返回用户的主要组
  • SecondaryGroupName：返回与之匹配的用户的第一个辅助组
  • Fixed ：返回规则中配置的队列名称
  • ApplicationType：返回应用程序类型（如果可用）

对于当前配置的修订版中的非托管队列，你不能提供任何特定于队列的属性。但是，将来我们应该考虑将特定的资源相关设置从受管父级传播到不受管子级，具体是：

  • 动态资源设置：

    • Guaranteed (resource)
    • Maximum (resource)

  • 动态运行应用程序限制：

    • Maximum (integer)

配置更新

更新队列定义将允许更新现有队列属性以及添加和删除队列。仅当可以解析配置时，新的队列定义才会变为活动状态。定义的更改是一种原子更改，将所有修改应用到一个动作中。

更新队列属性不会自动触发进一步的操作。这意味着，如果更改队列或其父级的最大资源数量，队列中的应用程序依然保持原样运行。调度程序将遵循新的属性值，随着时间的推移，这些属性值应会收敛。

如果将托管队列从配置中删除，则仅将其删除。在删除队列之前，它一定不能正在运行应用程序。这意味着从配置中删除托管队列时，它必须为空，否则系统需要允许队列耗尽。在删除队列之前强制将清空托管队列是不可能的，这意味着要分多个步骤删除托管队列：

  1. 队列已从配置中删除
  2. 队列被标记为 draining
  3. draining删除所有为空的托管队列

删除托管队列时，应妥善处理长时间运行的应用程序。调度程序应该跟踪并公开在很长时间内已处于draining状态的队列。在最佳情况下，队列更改应该通知应用程序以允许其释放资源。在所有情况下，都应通知队列管理员以使他们采取行动。当前，此操作是管理员将应用程序手动移动到其他队列。

调度程序会根据placement规则自动创建未在队列定义中定义的非托管队列。非托管队列的寿命与配置无关。每当非托管队列为空时，它将被删除。当新应用程序通过触发placement规则来请求队列时，将再次自动创建该队列。

删除空的托管或非托管队列由相同的删除代码处理，这些代码必须独立于配置更新和计划操作而运行。

配置会随着时间而改变。故障转移或重新启动的影响仍必须进行研究。基本要点：更改配置不会影响当前正在运行的应用程序。 应该以某种方式处理不存在的队列。

访问控制列表

调度程序ACL独立于队列ACL。默认情况下，不允许调度程序管理员提交应用程序或管理系统中的队列。

所有ACL类型都应使用相同的定义模式。我们应该至少允许使用可移植文件名字符集的POSIX用户名和组名。 但是，我们应该考虑到，我们可以根据系统运行的环境来设置域说明符(如HADOOP-12751所示)

默认情况下，访问控制处于启用状态，访问被拒绝。唯一的特殊情况是对于核心调度程序，它会自动将系统用户（调度程序进程所有者）添加到调度程序ACL。允许调度程序进程所有者确保进程所有者可以使用API调用任何管理操作。

访问控制列表可访问列表中指定的用户和组。它们不提供明确删除或拒绝访问列表中指定的用户和组的可能性。

访问控制列表定义为：

xxxxxxxxxx
ACL ::= “*” |  userlist [ “ “ grouplist ]
userlist ::= “” | user { “,” user }
grouplist ::= “” | group { “,” group }

此定义指定*的通配符，导致每个人都可以访问。如果用户列表为空而组列表为空，则没有人可以访问。这拒绝所有ACL有两种可能的表示形式：

  • 空的访问控制列表。
  • 一个空间。

如果没有配置访问控制列表，则默认情况下拒绝访问。
