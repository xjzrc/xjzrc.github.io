YuniKorn的抢占逻辑

要解决的问题：

根据我们从YARN Scheduler抢占中学到的经验教训。

这是最坏的事情：

  • 抢占是一把猎枪，而不是狙击手，当抢占决定做出时，没有人知道抢占的资源是否能满足queue/app/user。
  • 抢占逻辑和分配是不同的，我们必须实现（并模仿）在调度程序分配逻辑中所做的工作。

这是最重要的事情：

  • 抢占很快（由于使用了猎枪），回收数千个容器仅需1秒钟。
  • 我们已经了解处理DRF、多个抢占策略（队列间/队列内、散弹枪/外科抢占等）是多么痛苦，并且我们已经开发了一些良好的逻辑来确保更好的模块化和可插入性。

回答一些有关设计/实现选择的问题

1.我们真的想要抢占延迟吗？（或者我们只想控制节奏）

在CS中，我们具有抢占延迟功能，可以在抢占候选者中选择受害者，并等待一段时间才能杀死它。

抢占延迟的目的是：a.给应用程序一些必要的时间，以便他们可以准备一些不好的事情（不幸的是，至少从我所知的情况来看，没有应用程序可以对这些情况进行任何处理）。b。控制抢占速度。

在实践中，我发现它会引起很多问题，例如，当集群状态不断变化时，很难确保准确的抢占。

提案：移除抢占延迟,保持控制抢占速度的逻辑。(例如yarn.resourcemanager.monitor.capacity.preemption .total_preemption_per_round）。我们可以将分配与抢占一起进行。这并不意味着在发出抢占事件后容器将立即停止。相反，RM可以控制向容器发出信号和杀死容器之间的延迟。例如K8中POD的宽限终止:https://kubernetes.io/docs/concepts/workloads/pods/pod/#termination-of-pods

2.是要对每个调度逻辑进行抢占，还是可以定期进行？

在CS中，抢占逻辑会定期运行，例如每1秒或3秒运行一次。

由于抢占逻辑涉及一些繁重的逻辑，例如计算queues/apps.的份额。在进行精确的抢占时，我们可能需要扫描节点以寻找抢占候选者。考虑到这一点，我建议定期进行抢占。但是需要注意的重要一点是，我们需要尝试使用尽可能多的代码进行内部抢占，否则会有太多重复的逻辑，并且将来很难维护。

3.抢占成本和功能

我们发现增加抢占成本非常有帮助，例如容器使用时间，优先级，容器类型。它可以是成本函数（返回一个数字值），也可以是比较器（比较两个分配以进行抢占）。

for partition: 
  askCandidates := findAskCandidates(nAlloc, preemption=false)
  
  allocated, failed_to_allocated := tryAllocate(askCandidates);
  
  send-allocated-to-cache-to-commit;

  nAlloc -= len(allocated)  

抢占逻辑（在每个抢占周期调用）

xxxxxxxxxx
// It has to be done for every preemption-policy because calculation is different.
for preemption-policy: 
  preempt_results := policy.preempt()
  for preempt_results: 
     send-preempt-result-to-cache-to-commit;
     updated-missed-opportunity (allocated)

内部抢占策略：

inter-queue-preempt-policy:
  calculate-preemption-quotas;
  
  for partitions:
    total_preempted := resource(0);
    
    while total_preempted < partition-limited:
      // queues will be sorted by allocating - preempting
      // And ignore any key in preemption_mask
      askCandidates := findAskCandidates(N, preemption=true)
      
      preempt_results := tryAllocate(askCandidates, preemption=true);
      
      total_preempted += sigma(preempt_result.allocResource)
      
      send-allocated-to-cache-to-commit;
      
      update-missed-opportunity (allocated, failed_to_allocated);
      
      update-preemption-mask(askCandidates.allocKeys - preempt_results.allocKeys)
