kubectl 概述

kubectl是 K8s 集群的命令行工具，通过kubectl能够对集群本身进行管理，并能够在集群上进行容器化应用的安装部署。运行kubectl命令的语法如下所示：

xxxxxxxxxx
$ kubectl [command] [TYPE] [NAME] [flags]

这里的command，TYPE、NAME 和 flags 为：

  • command：指定要对资源执行的操作，例如create、get、describe和delete。

  • TYPE：指定资源类型，资源类型是大小学敏感的，开发者能够以单数、复数和缩略的形式，例如：

    xxxxxxxxxx
    $ kubectl get pod pod1 
    $ kubectl get pods pod1 
    $ kubectl get po pod1

  • NAME：指定资源的名称，名称也大小写敏感的。如果省略名称，则会显示所有的资源，例如:

    xxxxxxxxxx
     $ kubectl get pods

  • flags：指定可选的参数。例如，可以使用-s或者–server参数指定Kubernetes API server的地址和端口。

另外，可以通过在命令后面添加--help参数来获取更多的信息，例如：

xxxxxxxxxx
$ kubectl get nodes --help

检查 kubectl 是否已经配置好，并且能够访问集群，可以运行如下命令：

xxxxxxxxxx
$ kubectl version

部署应用

使用kubectl create 命令创建一个新的部署应用。创建时，我们需要提供部署名和应用的镜像地址（包含镜像在Docker hub仓库中的全路径）:

xxxxxxxxxx
$ kubectl create deployment kubernetes-bootcamp --image=gcr.io/google-samples/kubernetes-bootcamp:v1
deployment.apps/kubernetes-bootcamp created

使用get deployments命令列出所有的部署应用：

xxxxxxxxxx
$ kubectl get deployments
NAME                  READY   UP-TO-DATE   AVAILABLE   AGE
kubernetes-bootcamp   1/1     1            1           86s

在Kubernetes内部运行的Pods是在一个私有的、隔离的网络上运行的。默认情况下，它们在同一个kubernetes集群内的其他pod和服务中可见，但在该网络之外则不可见。我们可以使用kubectl命令通过一个API端点与应用程序进行交互。kubectl命令可以创建一个代理，将通信转发到集群范围的专用网络。代理可以通过按control-C终止，并且在运行时不会显示任何输出。

xxxxxxxxxx
$ kubectl proxy
Starting to serve on 127.0.0.1:8001

当代理生效后，我们的主机（在线终端）和Kubernetes集群就建立了连接。通过代理我们可以访问集群提供的API。例如，通过curl命令访问API，我们可以查询集群的版本：

探索应用

Kubernetes Pods

Pod是Kubernetes的抽象概念，它表示一组一个或多个应用程序容器（如Docker或rkt）以及这些容器的一些共享资源。这些资源包括：

  • 共享存储（Volumes）
  • 网络（一个唯一的集群IP地址）
  • 每个容器的运行信息，例如这个容器的镜像版本或者使用的特定端口

Nodes

Pod总是在Node上运行。Kubernetes集群中，Node是的工作节点，它可以是虚拟机，也可以是物理机，具体取决于集群。每个Node都由Master管理。一个Node中可以运行多个pod，Master 会根据每个节点上的可用资源自动处理集群节点之间的pod调度。

每个Node至少运行：

  • Kubelet，负责Master和Node之间通信的进程，它管理机器上运行的pod和容器。
  • 正在运行的容器（如Docker，rkt），负责从注册表中提取容器镜像、解压缩容器并运行应用程序。

暴露应用

Kubernetes Services

Pod实际上是有生命周期的。当工作节点死亡时，在该节点上运行的Pod也将丢失。然后，ReplicaSet可能会通过创建新Pod来动态地将群集重新达到所期望的状态，以保持应用程序继续运行。

Kubernetes中的Services是一种抽象，定义了Pod的逻辑集和访问Pod的策略。Services使从属Pod之间的松散耦合成为可能。像所有Kubernetes对象一样，使用YAML （首选）或JSON 定义服务。Services所针对的Pod集合通常由LabelSelector决定（请参见下文，了解为什么不包括selector在规范中就可能需要服务）。

尽管每个Pod都有一个唯一的IP地址，但是如果没有Services，这些IP不会暴露在群集外部。Services允许你的应用程序进行数据传输。通过type在ServiceSpec中指定可以以不同的方式暴露服务：

  • ClusterIP（默认）- 在群集的内部IP上公开服务，这种类型使得只能从群集内访问服务。
  • NodePort - 使用NAT在群集中每个选定节点的相同端口上公开服务。集群外部可以使用<NodeIP>:<NodePort>访问服务。ClusterIP的超集。
  • LoadBalancer -在当前云中创建一个外部负载平衡器（如果支持），并为该服务分配一个固定的外部IP。NodePort的超集。
  • ExternalName -externalName通过返回具有该名称的CNAME记录，使用任意名称（在规范中指定）公开服务。不使用代理。此类型需要v1.7或更高版本kube-dns。

Services可以控制Pod之间的数据传输。服务是允许Pod在Kubernetes中死亡和复制而又不影响应用程序的一种抽象。相互依赖的Pod（例如应用程序中的前端和后端组件）之间的发现和路由由Services来处理。

Labels

标签是附加在对象上的键/值对，可以以多种方式使用：

  • 指定用于开发，测试和生产的对象
  • 嵌入版本标签
  • 使用标签对对象进行分类

扩展应用

在前面的模块中，我们创建了一个Deployment，然后通过Service公开地将其暴露。部署仅创建了一个Pod来运行我们的应用程序。当流量增加时，我们将需要扩展应用程序以适应用户需求。

可以使用kubectl run命令的--replicas参数创建具有多个实例的Deployment

扩展部署将确保创建新Pod并将其调度到具有可用资源的节点上。Scala会将Pod的数量增加到新的所需状态。Kubernetes还支持Pods的自动缩放。缩放到零也是可能的，它将终止指定Deployment的所有Pod。

运行多个实例的应用程序需要将流量分配给所有实例。服务集成的负载平衡器会将网络流量分配给暴露的Deployment的所有Pod。服务将使用端点连续监视正在运行的Pod，以确保流量仅发送到可用的Pod。一旦运行了一个应用程序的多个实例，就可以进行滚动更新而无需停机。

更新应用

Rolling updates

用户期望应用程序始终可用，而开发人员则需要每天多次部署它们的新版本。在Kubernetes中，这是通过滚动更新来完成的。滚动更新允许通过用新的Pod实例增量更新Pod实例，从而在零停机时间内进行Deployment的更新。新的Pod将在具有可用资源的节点上安排。

与应用程序扩展类似，如果Deployment对外暴露，则该服务将在更新过程中仅将流量负载均衡到可用的Pod。可用的Pod是可供应用程序用户使用的实例。

滚动更新允许执行以下操作：

  • 将应用程序从一种环境升级到另一种环境（通过容器映像更新）
  • 回滚到以前的版本
  • 持续集成和持续交付应用程序，停机时间为零
