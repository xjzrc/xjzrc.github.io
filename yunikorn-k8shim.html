yunikorn-k8shim

yunikorn-k8shim 是一个定制化的 k8s 调度程序，它可以作为调度程序部署在K8s集群中。yunikorn-core封装了所有实际的调度逻辑，通过yunikorn-k8shim与K8s集群进行交互。默认情况下，如果pod的spec配置schedulerName: yunikorn属性，yunikorn-k8shim将处理所有pod的调度。

对 K8s Predicates的支持

Predicates是K8s中的一组预注册函数，调度程序调用这些函数来检查Pod是否适合分配给节点。通用Predicates有：节点选择器，pod亲和力/反亲和力等。为了在YuniKorn中支持这些Predicates，yunikorn-k8shim没有重新实现所有功能，而是尽可能重用K8s Predicates相关核心代码。

yuniKorn-core不了解RM的底层，因此Predicates功能在yunikorn-k8shim中以SchedulerPlugin的形式实现。SchedulerPlugin是一种plug/extend调度程序功能的方法。Shim可以实现此类插件并将其注册到yunikorn-core中，因此可以在yunikorn-core中调用插入的功能。目前yunikorn-core支持以下 SchedulerPlugins :

  • PredicatesPlugin : 运行一组特定的 Predicate 函数，以确定是否可以将提议分配到节点上。
  • VolumesPlugin : 确认分配后绑定Volume
  • ReconcilePlugin :重新同步调度程序缓存(将yunikorn-core端的缓存状态同步更新到shim端)，例如假定的分配。

工作流程

首先，RM需要将自己注册到yunikorn-core中，它会声明自己支持哪些调度程序插件接口。例如，RM可以实现PredicatePlugin接口并将其自身注册到yunikorn-core。然后，yunikorn-core将在做出分配决定之前调用PredicatePlugin API来运行Predicates 功能。

以下工作流程演示了当涉及PredicatePlugin怎么分配资源：

xxxxxxxxxx
pending pods: A, B
k8s-shim sends requests to yunikorn-core, including A, B
yunikorn-core starts to schedule A, B
  partition -> queue -> app -> request
    schedule A (1)
      run predicates (3)
        generate predicates metadata (4)
        run predicate functions one by one with the metadata
        success
        proposal: A->N
    schedule B (2)
      run predicates (calling shim API)
        generate predicates metadata
        run predicate functions one by one with the metadata
        success
        proposal: B->N
commit the allocation proposal for A and notify k8s-shim
commit the allocation proposal for B and notify k8s-shim
k8s-shim binds pod A to N
k8s-shim binds pod B to N

（1）和（2）并行运行。

（3）yunikorn-core调用一个schedulerPluginAPI来运行predicates，该API在k8s-shim端实现。

（4）K8s-shim根据当前的调度缓存生成元数据，该元数据包括一些有关节点和Pod的间歇状态。

Predicates 白名单

K8s-shim故意只支持Predicates白名单。主要是由于两个原因：

  • Predicates功能很耗时，会对调度程序性能产生负面影响。支持仅必要的Predicates可以最大程度地减少影响。这将可以通过CLI选项进行配置。
  • 尽管我们重用了一些单元测试，但实现很大程度上取决于K8的默认调度程序代码，但是覆盖范围仍然是一个问题。添加新Predicates时，将会继续提高覆盖范围。
