YuniKorn scheduler架构

以下是YuniKorn scheduler的核心组件：

  • Scheduler API Server: 负责RM和Scheduler之间的通信，它实现了 scheduler-interface GRPC 协议或仅仅是提供APIs服务。（用于进程内通信）
  • Scheduler Cache: 缓存与调度程序状态有关所有的数据，例如每个队列，节点， allocations的已用资源。不含调度程序的临时数据，例如预分配被抢占的资源，队列公平共享资源等 。
  • Scheduler Cache Event Handler: 处理所有需要更新调度程序内部状态的事件。因此，所有写操作都将被仔细处理。
  • Admin Service : 处理来自管理员的请求，该管理员还可以从存储中加载配置并更新调度程序策略。
  • Scheduler and Preemptor : 处理调度程序的内部状态(不属于Scheduler Cache)，例如内部保留等。 Scheduler 与Preemptor将协同工作，做出调度或抢占决策。

所有分配/抢占请求将由事件处理程序处理。

Scheduler 与 Preemptor 的实现

Scheduler的职责如下：

  • 根据队列之间的资源使用情况，对队列，应用程序进行排序，并找出应用程序分配的顺序。（这也将被抢占使用）。
  • 我们可能无法满足某些分配请求，我们需要跳过它们并找到下一个请求。
  • 由于资源碎片，某些分配请求可能无法满足。我们需要为此类请求保留空间。
  • 不同的节点可能属于不同的不相交的分区，我们可以使独立的调度程序运行
  • 在许多情况下，本地性仍然很重要，尤其是在本地情况下。
  • 能够配置和更改应用程序，队列的订购策略。
  • 应用程序可以选择自己的方式来管理节点种类。

Preemptor的职责如下：

  • 知道“谁想要资源”很重要，因此我们可以根据分配顺序进行抢占。
  • 进行抢占时，触发分配操作也很有效。考虑如何做。
  • 抢占需要注意队列资源平衡。
